# Deployment Compatibility Rules

These rules exist because the project deploys to a real EC2 instance
using an automated pipeline (user-data.sh → deploy-backend.sh →
deploy-frontend.sh) that pulls from Secrets Manager and generates
.env at deploy time. The deployed environment is NOT the same as your
local dev environment, and .env files are never committed to git.
Changes that work locally can silently break the live deployment if
these rules are not followed.

---

1. Never introduce a new required environment variable silently

---

If you add code that reads a new environment variable (via
process.env, a validation layer like Zod, or any config module):

- STOP before considering the task done.
- Report the new variable name, whether it's required or optional
  (with default), and what it's for.
- Do NOT assume it will "just be there" in production. It will not
  exist in the deployed .env unless it is explicitly added to:
  a) the AWS Secrets Manager secret (ecom-lite/prod/backend), and
  b) the .env-generation logic in deploy-backend.sh.
- Wait for confirmation that both have been updated before treating
  the feature as deployment-ready.

---

2. Never rename an existing environment variable without flagging it

---

Renaming, recasing, or restructuring how an env var is read (e.g.
introducing a validation/config layer that expects a different name
than before) is a deployment-breaking change even if it's cleaner
code. Any such rename must be reported explicitly, listing the OLD
name and the NEW name side by side, before it's considered complete.

---

3. Fail-fast validation layers require an explicit compatibility check

---

If you add or modify a startup-time validation layer (Zod schema,
config loader, etc.) that causes the app to refuse to start when a
variable is missing or malformed:

- You must explicitly state whether the CURRENT production .env
  (see the canonical variable list below) would pass or fail this
  validation as-is.
- If it would fail, you must state exactly why and what needs to
  change — do not leave this to be discovered at deploy time.

4. Canonical list of current production environment variables

---

Set in production .env (generated by deploy-backend.sh):

    PORT
    DATABASE_URL
    DB_SSL
    JWT_SECRET
    JWT_EXPIRES_IN
    NODE_ENV               (accepted, unused by env.js — stripped)
    FRONTEND_URL            (accepted, unused by env.js — stripped)
    SUPER_ADMIN_EMAIL       (added — required only if seed.js is
                              ever run manually)
    SUPER_ADMIN_PASSWORD    (added — same as above)
    SUPER_ADMIN_USERNAME    (added — defaults to 'admin' if absent)

Note: prisma/seed.js is NOT part of the automated deploy pipeline
(deploy-backend.sh only runs `prisma generate` and `prisma migrate
deploy`). Seeding is a deliberate manual step, run only when
bootstrapping a fresh database or recovering a SUPER_ADMIN account —
by design, not by omission. SUPER_ADMIN_EMAIL/PASSWORD are hard
requirements for seed.js and will crash it (exit 1) if absent when
manually invoked. 5. New routes intended for infrastructure use must stay dependency-free

---

Routes used for deployment health checks (e.g. GET /health) must
never depend on the database, external services, or authentication.
They must return 200 purely based on "the process is alive and
responding." If you change what /health does, or add a new
infrastructure-facing route, flag it explicitly — the deploy scripts
depend on this route's behavior for automated verification.

---

6. Do not assume special characters are absent from secrets

---

Passwords, tokens, and secrets in Secrets Manager may contain
characters like $, @, :, /, %, or quotes. Any script or code path
that constructs connection strings, writes .env files, or interpolates
secret values must not assume these characters are absent. In
particular:

- Never use an unquoted heredoc (<<EOF) to write secret values into a
  file — bash will re-expand $ sequences and can silently truncate or
  corrupt the value. Use a quoted heredoc (<<'EOF') with placeholder
  substitution via sed, or an equivalent safe method.
- If a value must be embedded in a URL (e.g. inside DATABASE_URL),
  URL-encode it.

---

7. Report deployment risk explicitly, ranked

---

When summarizing a batch of changes (a "walkthrough" or implementation
report), explicitly separate:

- Changes that could prevent the app from starting on EC2 (env var
  changes, startup validation changes, new required infra dependencies)
- Changes that could cause functional regressions at runtime (shared
  validators, refactored business logic, changed response shapes)
- Changes with no runtime impact (docs, dev-only files, deleted stubs)

Do not bury deployment-breaking changes in the middle of a long list
alongside documentation updates. Lead with them, clearly labeled, the
way you would flag an Architecture Deviation.

---

8. When in doubt, ask before the task is marked complete

---

If you're unsure whether a change affects what EC2 needs, treat it as
if it does. State your uncertainty explicitly rather than assuming it's
fine. This matches the project's existing "STOP, Report, Approve,
Continue" rule for architecture changes — deployment compatibility is
treated the same way.
